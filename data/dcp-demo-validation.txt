# DCP Pipeline Validation — Demo Reasoning
# Task: Implement a retry wrapper for OpenAI API calls with exponential backoff

## LOGIC DRAFT
Primary algorithmic path: Wrapper function attempts API call → on failure (rate limit/network), sleeps with exponential backoff (1s, 2s, 4s, 8s) → retries up to 5 times → returns Result<T, Error>.

Invariants:
- Total retries ≤ 5
- Backoff delay = 2^(attempt-1) seconds, capped at max_backoff
- Rate limit errors (HTTP 429) always trigger retry
- Network timeout errors always trigger retry
- Non-retryable errors (HTTP 4xx except 429, 5xx except 503) don't retry
- Result type is Result<T, RetryError> where RetryError wraps the last error
- Maximum total wait time ≤ 31s (1+2+4+8+16)

Inputs: API call closure F: Fn() -> Result<T, ApiError>, max_retries: u32 = 5, base_delay: Duration = 1s
Outputs: Result<T, RetryError>

## EDGE-CASE CRITIC
Failure 1: Zero retries (max_retries = 0) → should be rejected at compile time or return immediately without retry.
Mitigation: Use NonZeroU32 for max_retries or enforce minimum of 1.

Failure 2: Overflow in delay calculation (2^63 seconds) → Duration::from_secs() panics on overflow.
Mitigation: Cap delay at MAX_BACKOFF (e.g., 60s) using min(calculated, max_cap).

Failure 3: The closure mutates external state → retrying a non-idempotent operation could double-submit.
Mitigation: Document the requirement that the closure must be idempotent. Return `RetryError::NonIdempotent` if state is detected.

Failure 4: Thread panic inside closure during retry → retry loop exits without proper cleanup.
Mitigation: Use std::panic::catch_unwind to wrap the closure call.

Failure 5: Rate limit Retry-After header exceeds max_backoff → should respect server's specified delay but cap at a maximum.
Mitigation: Parse Retry-After header, use max(backoff_delay, retry_after), then cap at overall max.

## AST PROOF

Step 1: attempt = 1, delay = 1s
  → Call API → HTTP 429 (rate limit)
  → Is retryable? Yes (429)

  → errors.push(ApiError::RateLimit)
  → sleep(1s)
State after Step 1: attempt=1, delay=1, errors=[RateLimit]

Step 2: attempt = 2, delay = 2s
  → Call API → Network timeout
  → Is retryable? Yes (timeout)
  → attempt < max_retries? (2 < 5) Yes

  → sleep(2s)
State after Step 2: attempt=2, delay=2, errors=[RateLimit, Timeout]

Step 3: attempt = 3, delay = 4s
  → Call API → Success! Response: {"choices": [...]}
  → Return Ok(response)
Final state: Ok(response), 3 attempts, 2 errors logged, total wait = 3s

- Total retries (2) ≤ 5 ✓
- Backoff followed: 1s, 2s ✓ (was 2^0=1, 2^1=2 -- correct)
- Non-retryable errors: none encountered ✓
